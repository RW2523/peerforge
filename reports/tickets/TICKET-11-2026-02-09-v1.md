  - allowed_participant_ids array
  - granted_by, granted_at, expires_at

- ✅ Preview generation (chunk count, topics, date range)
- ✅ Edge cases (6 scenarios: deleted source, revoke mid-debate, circular refs, cross-workspace, etc.)

- ✅ Non-goals (cross-workspace, auto-suggestions, summarization, decay)

**Key Architectural Decisions Implemented:**
- Relational join table (`debate_memory_grants`) from day 1 for auditability
- User-controlled, explicit imports (not automatic)
- Granular agent scoping (all agents or specific agents)
- Immutable after debate starts (no mid-debate revocation)
- Soft delete for source debates with active grants

## Spec Quality Assessment

### Live Artifacts Spec

**Strengths:**
- Comprehensive API contract definitions (request/response examples)
- Clear data model reusing existing tables (no new storage systems)
- Block types have validation rules and rendering requirements
- Quality checks have implementation pseudocode
- Coherence workflow addresses cost and user control
- 4 built-in templates ready for implementation

**Potential Engineering Questions:**
1. SSE event ordering guarantees (answer: use event timestamps, client sorts)
2. Max artifact size (answer: soft limit 50 sections, 10k words total)
3. Chart rendering on backend for export (answer: use headless Chrome or Chart.js SSR)

---

### Memory Import Spec

**Strengths:**
- Clear UI wireframes for setup flow
- Join table design is auditable and performant
- Enforcement rules have SQL examples
- Preview generation detailed
- Compliance export addressed

**Potential Engineering Questions:**
1. How to extract topics for preview (answer: keyword extraction or lightweight LLM call)
2. Performance of grant checks during retrieval (answer: indexed, <50ms target)
3. Soft delete implementation for debates (answer: add `deleted_at` column, filter in queries)
