## Implementation Details

### Workspace Resolution (Option A)
**Backend strategy:**
1. Extract `user_id` from JWT (`sub` claim)
2. If JWT contains `workspace_id` claim → use it directly
3. If JWT does NOT contain `workspace_id` → query `user_workspaces` table
4. Return first workspace found for user
5. If no workspace → return 403

**Frontend strategy:**
- All API requests include `Authorization: Bearer <supabase_access_token>`
- Token obtained from Supabase session via `getAccessToken()`
- No dev/test token fallback (production-only mode)

### Auth Flow
1. User visits `/login`
2. Enters email/password OR requests magic link
3. Supabase creates session with JWT access_token
4. All API requests include token in Authorization header
5. API validates JWT signature and resolves workspace
6. User can access debates in their workspace(s)

### JWT Validation
**Settings:**
- `verify_exp`: True (expiration checked)
- `verify_iat`: False (disabled to avoid clock skew in tests)
- `algorithms`: ['HS256']
- Secret: `SUPABASE_JWT_SECRET` from env

**Note:** In production, consider enabling iat verification and adding proper clock sync.
