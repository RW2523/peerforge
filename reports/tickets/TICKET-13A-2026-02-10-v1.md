## Summary

Implemented the backend "Preflight Core" that prepares each agent BEFORE the debate starts by:
- Gathering context from materials and imported memory (via grants)
- Generating a prep pack memo for each agent
- Storing prep packs as `agent_knowledge_units` with `knowledge_type='prep_pack'`
- Tracking per-participant progress with retry/skip support
- Ensuring audit compliance (logs memory retrieval with grant_ids)

**Result**: Agents can now enter debates fully prepared with relevant context, and the system tracks all preparation steps for audit and reliability.

## What Works Now (Plain English)

1. **Start Preflight for a Debate**
   - Call `POST /debates/{debate_id}/preflight/start`
   - System creates a run and enqueues preparation tasks for each participant
   - Returns run_id and initial participant statuses

2. **Monitor Progress**
   - Call `GET /debates/{debate_id}/preflight/status`
   - See overall run status (queued/running/completed/failed)
   - See per-participant status with prep_pack_knowledge_id when ready

3. **Handle Failures**
   - If a participant fails, call `POST /debates/{debate_id}/preflight/retry` with participant_id
   - System re-attempts preparation
   - Failure reason captured in error field

4. **Skip Participants**
   - Call `POST /debates/{debate_id}/preflight/skip` with participant_id and reason
   - System marks as skipped with audit trail
   - Debate can proceed with partial preflight

5. **Prep Packs Are Stored**
   - Each successful preflight creates an `agent_knowledge_units` row
   - knowledge_type='prep_pack'
   - Links to agent_id and source_debate_id
   - Metadata includes: materials reviewed, grants used, model used

## Architecture

### Data Flow

```
POST /preflight/start
  ↓
Create preflight_runs row (status=queued)
  ↓
Create preflight_participant_runs rows (one per participant, status=queued)
  ↓
Enqueue Celery task: orchestrate_preflight(run_id, debate_id)
  ↓
For each participant:
  ├── Update participant_run status → running
  ├── Resolve agent_id from participant.agent_config
  ├── Gather context:
  │   ├── Materials chunks (memory_chunks where source_debate_id=current)
  │   └── Imported memory chunks (retrieve_allowed_chunks with grants)
  ├── Build prep prompt (problem + materials + imported context)
  ├── Generate memo (OpenRouter call or placeholder)
  ├── Store as agent_knowledge_units (knowledge_type='prep_pack')
  ├── Update participant_run status → success
  └── Link prep_pack_knowledge_id
  ↓
Update preflight_runs status → completed
```

### Memory Integration

Preflight uses the same memory enforcement as runtime retrieval:
- Calls `retrieve_allowed_chunks(debate_id, participant_id, query, top_k)`
- Respects debate_memory_grants (all_agents vs specific_agents)
- Logs to memory_access_log with grant_ids
- Participant only sees context they're granted

This ensures:
- Consistent access control (setup grants → preflight → runtime)
- Full audit trail (who accessed what, when, via which grants)
- No duplicate enforcement logic

### Retry/Skip Model

**Retry** (for transient failures):
- Reset participant_run to queued
- Clear error field
- Re-execute task
- Use case: Network timeout, temporary OpenRouter API issue

**Skip** (for permanent blocks):
- Mark participant_run as skipped
- Record skip_reason (human-readable)
- Set completed_at
- Use case: Agent misconfigured, model unavailable, user decision to proceed without

Both actions are auditable via preflight_participant_runs table.

## Known Behaviors (V1)

### OpenRouter Key Requirement

**V1 Behavior**: If no OpenRouter key is in `policy_config`, preflight generates a placeholder prep pack with:
- Problem statement
- Materials count
- Imported chunks count
- Message: "Placeholder prep pack - OpenRouter key required for real generation"

**Production**: Setup wizard should capture OpenRouter key and store in policy_config (or pass separately). This will be addressed in TICKET-13B (UI integration).

### No Real Research Calls

**V1**: Research provider interface is NOT implemented in this ticket. All prep packs are generated from:
- Problem statement
- Materials chunks
- Imported memory chunks

**Future** (TICKET-13B): Add ResearchProvider interface + Perplexity integration for internet research (policy-controlled, citations required).

### Synchronous Orchestration

**V1**: Participants are processed sequentially in the orchestrator.

**Why**: Simplicity, easier debugging, predictable resource usage.

**Future**: Can use Celery group/chord for parallel execution if performance is needed.

### No State Machine Integration

**V1**: Preflight does NOT update debate.state (e.g., to 'preparing_agents' or 'ready').

**Why**: Debate state machine updates were deferred to keep scope manageable.

**Future** (TICKET-13C): Add state machine integration:
- pending → materials_processing → materials_ready → preparing_agents → ready → running

## Engineering Notes

### Why Placeholder Prep Packs Work for V1

**Problem**: Tests need to run without real OpenRouter calls.

**Solution**: If `policy_config.openrouter_key` is missing, generate a structured placeholder that:
- Includes problem statement
- Lists materials count and imported chunks count
- Shows agent config (model_id, system_prompt)
- Is valid for testing status/retry/skip logic

**Production**: UI will capture OpenRouter key during setup, and real prep packs will be generated.

### Why agent_knowledge_units (Not a New Table)

**Decision** (from DECISIONS-SOURCE-OF-TRUTH): Reuse existing tables, no duplicate storage systems.

**Benefits**:
- Consistent schema for all agent knowledge (artifacts, prep_packs, research)
- Unified retrieval interface (can query all knowledge by agent_id)
- Simpler migrations

**Implementation**:
- Use `knowledge_type='prep_pack'` to distinguish from other knowledge
- Link to source_debate_id for scoping
- Metadata stores provenance (materials_count, grants_used)

### Why Synchronous Orchestration is OK for V1

**Alternatives Considered**:
1. Parallel (Celery group): Faster but harder to debug
2. Sequential (current): Simpler, predictable, easier retry logic

**V1 Choice**: Sequential.

**Why**:
- Typical debate has 3-5 participants (< 30s total prep time)
- Easier to reason about failures
- Predictable resource usage (1 OpenRouter call at a time)

**Future**: If debates scale to 20+ participants, use Celery group.
