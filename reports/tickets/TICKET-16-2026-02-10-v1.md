## Known Limitations (V1)

### Specific Participants Mapping (Partial Support)

**Issue**: When a user selects "Only Selected Participants" and checks specific participants by index, the UI cannot yet map those indices to actual `participant_ids` because `POST /debates/setup` does not return the created `participant_ids`.

**V1 Behavior**:
- UI allows selecting specific participants
- Backend accepts `participant_ids` array in import request
- But `handleLaunch` only supports `scope: 'all_agents'` fully
- Comment in code notes this limitation

**Workaround for V1**:
- Use "All Participants" scope (fully functional)
- "Only Selected Participants" can be selected but won't map correctly until backend returns participant_ids

**Future Fix** (TICKET-16.1 or backend enhancement):
- Update `POST /debates/setup` response to include `participant_ids`
- Update `handleLaunch` to map selected indices to IDs
- Then `scope: 'specific_agents'` will work end-to-end

### Preview Uses Static Stats

**Issue**: The preview panel shows chunk counts from the importable source's metadata (from GET /workspaces/.../memory/importable), but does not call GET /debates/.../memory/preview with a target debate_id (because debate isn't created yet).

**V1 Behavior**:
- Preview shows: total_chunks, material_count, artifact_count from source metadata
- This is sufficient for V1 (user sees what exists in the source)

**Future Enhancement**:
- Call preview endpoint after debate creation but before grants
- Show more detailed breakdown (by source_type, date ranges)
- Show "what will participants actually see" preview

## Engineering Notes

### Why useMemoryImport Hook?

**Problem**: Setup page was 324 lines, exceeding 300-line limit for UI components.

**Solution**: Extracted memory import state and logic into `useMemoryImport` hook (54 lines):
- Manages `MemoryImportConfig` state
- Provides `validateMemoryImport()` for pre-submission checks
- Provides `createMemoryGrants()` for grant creation with error handling
- Returns clean interface for setup page

**Result**: Setup page reduced to 293 lines (under limit), logic reusable.

### Why Memory Import is Optional

**Design Decision**: Memory Import defaults to OFF and is entirely optional because:
- Many debates don't need prior context (new topics, different teams)
- Users should explicitly opt-in to context sharing (privacy/security)
- Simpler default flow reduces friction for first-time users
- Enabled state shows clear value prop ("share insights from past meetings")

### Grant Creation Timing

**Why create grants immediately after setup**:
- Grants are metadata (lightweight, fast)
- No chunking/embedding work required
- Fail-fast: user knows immediately if something went wrong
- Can retry or proceed without grants

**Why not defer to first retrieval**:
- Audit integrity: grants should exist before debate starts
- UI can show grants in debate metadata
- Enforcement hook can validate grants exist
