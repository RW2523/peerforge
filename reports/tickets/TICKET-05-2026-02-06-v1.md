## Implementation Details

### Architecture

**Request Flow:**
1. Client sends `POST /debates/run` with problem_statement, 3 agents, OpenRouter key
2. FastAPI endpoint validates request (Pydantic models)
3. DebateEngine initializes with BYOK OpenRouter key
4. Engine creates debate + participants in database
5. Engine executes 5-turn round-robin loop:
   - Turn 1: Agent 0
   - Turn 2: Agent 1
   - Turn 3: Agent 2
   - Turn 4: Agent 0
   - Turn 5: Agent 1
6. Each turn calls OpenRouter API, persists event to DB
7. Final LLM call generates summary + minutes + action items
8. Debate marked as "completed" in DB
9. Returns JSON with debate_id, outputs, event_history

**Database Operations (11 total):**
- 1 debate INSERT
- 3 participant INSERTs
- 1 system event INSERT
- 5 agent message event INSERTs
- 1 debate UPDATE (mark completed)

**OpenRouter Integration:**
- BYOK (Bring Your Own Key) per request
- No stored credentials
- Retry logic with exponential backoff (max 2 attempts)
- 120s timeout per request
- Error handling: 401 for auth errors, 500 for API errors

### Key Components

**`src/openrouter_client.py` (99 lines):**
- `OpenRouterClient` class with chat_completion method
- Retry decorator from tenacity (2 attempts, exponential backoff)
- Handles 401/403 auth errors, timeouts, request errors
- Enforces OpenRouter-only policy (no direct provider SDK usage)

**`src/debate_engine.py` (212 lines):**
- `DebateEngine` class with run_debate method
- Deterministic 5-turn round-robin logic (`turn % 3` for agent selection)
- DB persistence for debate, participants, events

## Known Limitations

### M1 Scope Constraints:
1. **Fixed 5-turn limit:**
   - **Limitation:** Debate always runs exactly 5 turns, no dynamic length
   - **Impact:** Cannot adjust debate length based on convergence or complexity
   - **Mitigation:** M2 will add manual end control

2. **No realtime streaming:**
   - **Limitation:** Client must wait for full 5-turn completion (30-120s depending on models)
   - **Impact:** No progress updates during execution
   - **Mitigation:** M2 will add SSE/WebSocket for realtime token streaming

3. **No intervention controls:**
   - **Limitation:** Cannot pause, resume, or intervene mid-debate
   - **Impact:** User cannot steer discussion once started
   - **Mitigation:** M2 will add pause/resume and intervention endpoints

4. **Basic summary generation:**
   - **Limitation:** Summary quality depends on LLM's ability to output JSON
   - **Impact:** Non-structured responses fall back to generic action items
   - **Mitigation:** JSON parsing has fallback; future iterations can use structured output APIs

5. **Hardcoded demo workspace/tenant:**
   - **Limitation:** Uses seed data IDs (`00000000-0000-0000-0000-000000000101`)
   - **Impact:** Single-tenant for M1; all debates in same workspace
   - **Mitigation:** M3 will add multi-tenant support

### Performance Considerations:
- **Latency:** 30-120 seconds for 5-turn debate (model-dependent)
- **Cost:** 6 OpenRouter API calls per debate (5 turns + 1 summary)
- **Concurrency:** No rate limiting implemented yet (future: per-tenant limits)
