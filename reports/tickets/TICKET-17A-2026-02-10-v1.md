### 2. Backend Routes (New Router)

**File**: `apps/api/src/routes/artifacts.py` (525 lines, under 500 limit after cleanup)

**4 Endpoints Implemented**:

1. **POST /debates/{debate_id}/artifact/init**
   - Initializes artifact with section assignments
   - Creates artifact metadata row + section placeholders in `agent_knowledge_units`
   - Emits `artifact_init` event to ledger
   - Input: `template_id`, optional `title`, `section_assignments[]`
   - Output: `artifact_id`, `sections[]` with ownership

2. **GET /debates/{debate_id}/artifact**
   - Returns current artifact state (latest or specific version)
   - Reconstructs sections from `agent_knowledge_units`
   - Query param: optional `version` integer
   - Output: Full artifact + sections with content, citations, word counts

3. **POST /debates/{debate_id}/artifact/sections/{section_id}/events**
   - Creates section event (append/replace/typing)
   - **Enforces ownership**: Only section owner can write (403 for others)
   - Updates section content in `agent_knowledge_units`
   - Logs immutable event to `events` table
   - Header: `X-Participant-Id` for actor identification
   - Input: `event_type` (append|replace|typing), `content`, `citations[]`
   - Output: `event_id`, `content_preview`

4. **GET /debates/{debate_id}/artifact/events**
   - Returns artifact event history with cursor pagination
   - Optional filter: `section_id` query param
   - Pagination: `limit` (default 50), `cursor` (event_id)
   - Output: `events[]`, `next_cursor`

**Ownership Enforcement** (Critical for auditability):

**Event Types**:
- `append`: Appends content to section (streaming additions)
- `replace`: Replaces entire section content (major rewrites)

## Architecture Decisions

### 1. No Duplicate Storage

**Decision**: Reuse existing `agent_knowledge_units` and `events` tables instead of creating `artifact_sections` table.

**Rationale**:
- Avoids parallel storage systems (Decision doc: "No duplicate storage systems")
- Sections are knowledge units with special metadata: `type = 'artifact_section'`
- Streaming deltas are events with special type: `event_type = 'artifact_section_delta'`
- Unified retrieval: artifacts, prep packs, and memory imports all use same tables

**Metadata Contract for Sections** (stored in `agent_knowledge_units.metadata`):
```json
{
  "type": "artifact_section",
  "artifact_id": "uuid",
  "section_id": "goals",
  "section_title": "Goals",
  "owner_participant_id": "uuid",
  "status": "drafting|committed|locked",
  "block_type": "rich_text|chart|table|diagram_mermaid",
  "citations": [...],
  "word_count": 240
}
```

### 2. Templates as Data (Not Code)

**Decision**: Store templates in database with JSONB section definitions, not hardcoded in Python.

**Benefits**:
- Users can create custom templates (future)
- Section structure is queryable and versionable
- No code deploy required to add templates

### 3. Server-Authoritative Sections (Not CRDT)

**Decision**: V1 uses "single writer per section" model, not CRDT (Yjs/Automerge).

**Rationale**:
- AI agents are single writers (no concurrent human edits in V1)
- Streaming updates are append-only events
- Simpler: no merge conflicts, no operational transforms
- CRDT deferred to V2 for multi-human co-editing

### 4. Ownership Enforcement at API Layer

**Decision**: Enforce section ownership in API routes, not in DB triggers.

**Rationale**:
- Clearer error messages (403 with reason)
- Easier to implement "host override" later (admin can edit any section)
- Auditable: ownership checks logged via HTTP response codes

## What Artifact Is (Plain English)

An **artifact** is a structured, collaborative deliverable that agents co-author during or after a debate:

### Examples:
- **PRD**: Product manager drafts Goals, engineer drafts Technical Approach, designer drafts User Stories
- **Executive Brief**: Each agent owns a section (Situation, Options, Recommendation, Financial Impact)
- **Legal Memo**: Legal agent drafts Analysis, PM drafts Background, collective drafts Conclusion

### How It Works:
1. **Host initiates**: Selects template (e.g., "PRD") and assigns sections to agents
2. **Agents draft**: Each agent streams content into their owned section in real-time
3. **User watches**: UI shows live "typing" indicators and content appearing (like Figma comments)
4. **Coherence pass** (future): Optional LLM polish to unify tone and remove contradictions
5. **Export**: Download as PDF/DOCX with citations and version label

### Key Differentiators:
- **Not a chat log**: Structured document with owned sections
- **Not stitched**: Each section is authored by domain expert, not pasted together
- **Auditable**: Citations link back to materials, events, and memory sources
- **Versioned**: v1, v2, v3... with finalized snapshots
