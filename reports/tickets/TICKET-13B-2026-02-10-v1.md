## Summary

Implemented the user-facing preflight preparation UI that integrates with the TICKET-13A backend. Users can now:
- Start preflight preparation for all agents from the setup wizard
- Monitor per-agent progress in real-time (queued/running/success/failed/skipped)
- Retry failed preparations or skip with reason (auditable)
- Preview prep pack content when ready
- Enter the room only after preflight completes or explicit skip/override

**Result**: Complete setup-to-room flow with agent preparation, progress tracking, error handling, and quality controls.

## Technical Architecture

### Polling Strategy

**usePreflight Hook**:
- Starts polling when `startPreflight()` called
- Polls `getPreflightStatus()` every 2 seconds
- Stops polling when:
  - Run status = 'completed' or 'failed'
  - Component unmounts (cleanup)
- Uses `useRef` to track current debate_id and interval

**Why 2 seconds?**
- Fast enough for responsive UI (users see updates quickly)
- Slow enough to avoid excessive API load
- Typical prep time: 5-15 seconds per agent (2-7 polls per agent)

### State Management

**Preflight State**:
- `status`: Full PreflightStatusResponse from backend
- `isPolling`: Boolean, true while actively polling
- `error`: String | null, user-friendly error messages
- `isStarted`: Computed, true if preflight run exists
- `isCompleted`: Computed, true if run status terminal
- `canContinue`: Computed, true if all agents ready or explicitly skipped
- `hasFailures`: Computed, true if any agent failed
- `readyCount / totalCount`: Computed for progress bar

**Setup Page State**:
- `createdDebateId`: Set after debate creation, passed to PreflightStep
- `createdParticipantIds`: Set after debate creation, used for mapping UI participants to backend participant_ids
- Step 5 (Preflight) cannot navigate back (debate already created)

### Status Mapping

Backend → UI:
```
Backend Status     UI Pill          Color      Actions
──────────────     ───────          ─────      ───────
queued           → "Queued"         Gray       Skip
running          → "Preparing..."   Blue       Skip (pulse animation)
success          → "Ready"          Green      View prep pack
failed           → "Failed"         Red        Retry, Skip
skipped          → "Skipped"        Yellow     (show reason)
```

### Participant Identity Resolution

**Challenge**: UI has participant array indices, backend needs participant_ids.

**Solution** (from TICKET-16.1):
- `handleCreateDebate` calls `api.setupDebate()` → returns `participant_ids[]`
- Store in `createdParticipantIds` state
- Pass to `PreflightStep` as prop
- `getParticipantName(participantId)`: maps participant_id → UI participant array index → name/role

**Why this works**:
- Participant order is preserved: `api.setupDebate` creates participants in order
- `result.participant_ids[0]` corresponds to `participants[0]` in UI
- PreflightStep receives both arrays and can map correctly

## Design Decisions

### Why Preflight is Step 5 (Not After Review)?

**Decision**: Insert Preflight between Memory Import (step 4) and Review (step 6).

**Rationale**:
- User must commit to debate before preparation starts (debate_id required)
- Preparation is expensive (time + potential OpenRouter calls)
- Cannot allow "go back" after debate created (would orphan debate)
- Review step is optional/passive; Preflight is active/blocking

**Alternative Considered**: Review → Preflight → Launch
- **Rejected**: Creates confusing flow (review, then prepare, then review again?)

### Why No "Skip All" Button?

**Decision**: Require individual skip with reason per agent.

**Rationale**:
- Forces user to acknowledge reduced quality per agent
- Audit trail: each skip has reason, not bulk action
- Prevents accidental degradation of entire debate

**Future**: If enterprise users request "skip all for testing", add with explicit warning + policy control.

### Why Extract Dialogs Into Separate File?

**Decision**: Split PreflightStep into PreflightStep.tsx (289 lines) + PreflightDialogs.tsx (101 lines).

**Rationale**:
- Obey 300-line file size limit (quality gate)
- Dialogs are pure presentation, no business logic
- Easier to test/maintain when isolated

**Alternative Considered**: Keep dialogs inline
- **Rejected**: Would exceed 300 lines (gate failure)

### Why Poll Instead of WebSocket?

**Decision**: Use HTTP polling (every 2s) instead of WebSocket/SSE for preflight status.

**Rationale** (V1):
- Simpler implementation (no WebSocket client code)
- Preflight is short-lived (< 1 minute typical)
- SSE already used for debate events; separate preflight SSE adds complexity
- 2s polling is acceptable for this use case (not real-time chat)

**Future** (TICKET-13C): Add SSE for preflight events if user feedback shows need.

## Known Behaviors (V1)

### Prep Pack Preview is Minimal

**V1 Behavior**: "View prep pack" shows metadata preview (chunks processed, grants used), not full memo content.

**Why**: 
- Backend returns `prep_pack_knowledge_id` in status response
- Full content is in `agent_knowledge_units.content` (not fetched by status endpoint)
- Adding GET endpoint for prep pack content was deferred to keep scope tight

**Future** (TICKET-13B.1): Add `GET /preflight/prep-pack?participant_id=...` endpoint to fetch full content.

### Preflight Does NOT Update Debate State

**V1 Behavior**: Debate remains in 'pending' state before and after preflight.

**Why**:
- State machine integration deferred to TICKET-13C
- Room can start regardless of preflight (user can skip all agents if needed)

**Future** (TICKET-13C): Add states: `materials_processing` → `materials_ready` → `preparing_agents` → `ready`.

### No Automatic Preflight Start

**V1 Behavior**: User must click "Start preparation" button.

**Why**:
- Explicit user action required (preparation costs time/money)
- User might want to review participants first
- Clear separation: debate creation ≠ preflight start

**Alternative Considered**: Auto-start on step 5 entry
- **Rejected**: Less control, unexpected behavior

### Prep Pack Content is Placeholder (No OpenRouter Key)

**V1 Behavior** (from TICKET-13A): If no `openrouter_key` in `policy_config`, preflight generates placeholder prep pack with:
- Problem statement
- Materials count
- Imported chunks count
- Message: "Placeholder prep pack - OpenRouter key required for real generation"

**Production**: Setup wizard should capture OpenRouter key and store in policy_config (or pass separately).

**Future** (TICKET-13B.2): Add OpenRouter key input in setup wizard or settings, pass to setupDebate.
