### Step 4: Embeddings Generation

**API Call** (user-triggered or automatic):
```http
POST /debates/{debate_id}/materials/{material_id}/embed
Headers:
  Authorization: Bearer {supabase_jwt}
  X-OpenRouter-Key: sk-or-v1-...
```

**Processing** (for each chunk):
1. Call OpenRouter embeddings API
2. Store vector in `memory_chunks.embedding_vector` (JSONB)

### Step 5: Preflight with Semantic Retrieval

**API Call** (user clicks "Start Preflight"):
```http
POST /debates/{debate_id}/preflight/start
Headers:
  Authorization: Bearer {supabase_jwt}
  X-OpenRouter-Key: sk-or-v1-...
```

**Backend Flow** (per participant):
1. **Query Embedding Generation** (BYOK-safe):
   - Build semantic query: `{problem_statement} + Role: {system_prompt}`
   - Call OpenRouter embeddings API (key in-memory only)
   - Store embedding vector in `preflight_participant_runs.metadata`
   - Drop key (Python GC)

2. **Semantic Retrieval**:
   - Load chunks with `embedding_status='complete'`
   - Compute cosine similarity (Python)
   - Return top 15 chunks
   - Record `retrieval_mode='semantic'` in metadata

3. **Prep Pack Generation**:
   - Call OpenRouter chat API with context
   - Store memo in `agent_knowledge_units` (knowledge_type='prep_pack')

**Status Check**:
```http
GET /debates/{debate_id}/preflight/status
```

**Response**:
```json
{
  "run_id": "run123",
  "status": "success",
  "participant_runs": [
    {
      "participant_id": "p1",
      "status": "success",

      "metadata": {
        "retrieval_mode": "semantic",

        "imported_chunk_ids": [],
        "embeddings_used": true,
        "query_embedding_model_id": "moonshot/kimi-embeddings-v1"
```

## Deployment Readiness

### ✅ Production-Ready Features
1. **Auth**: Supabase JWT validated on all protected endpoints
2. **BYOK**: OpenRouter keys never stored server-side (sessionStorage only)
3. **Realtime**: SSE works with auth (fetch-based, not EventSource)
4. **Semantic**: Embeddings + cosine similarity retrieval operational
5. **Audit**: All actions logged with sequence numbers + provenance

### ⚠️ Known Limitations (Acceptable for V1)
1. **Presence timeout**: No auto-offline after N minutes (manual leave only)
2. **Typing debounce**: 1.5s throttle (acceptable, prevents spam)
3. **Participant names**: IDs shown in presence (human names require mapping)
4. **OCR**: Baseline only (Tesseract stub, not vision models)
5. **Vector search**: Python cosine (works, but pgvector migration pending for scale)
